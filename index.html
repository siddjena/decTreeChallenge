The Core Problem: Decision trees are often praised for their interpretability and ability to handle both numerical and categorical variables. But what happens when we encode categorical variables as numbers? How does this affect our understanding of feature importance?

What is Feature Importance? In decision trees, feature importance measures how much each variable contributes to reducing impurity (or improving prediction accuracy) across all splits in the tree. It‚Äôs a key metric for understanding which variables matter most for your predictions.

ImportantüéØ The Key Insight: Encoding Matters for Interpretability

The problem: When we encode categorical variables as numerical values (like 1, 2, 3, 4‚Ä¶), decision trees treat them as if they have a meaningful numerical order. This can completely distort our analysis.

The Real-World Context: In real estate, we know that neighborhood quality, house style, and other categorical factors are crucial for predicting home prices. But if we encode these as numbers, we might get misleading insights about which features actually matter most.

The Devastating Reality: Even sophisticated machine learning models can give us completely wrong insights about feature importance if we don‚Äôt properly encode our variables. A categorical variable that should be among the most important might appear irrelevant, while a numerical variable might appear artificially important.

Let‚Äôs assume we want to predict house prices and understand which features matter most. The key question is: How does encoding categorical variables as numbers affect our understanding of feature importance?

The Ames Housing Dataset üè†

We are analyzing the Ames Housing dataset which contains detailed information about residential properties sold in Ames, Iowa from 2006 to 2010. This dataset is perfect for our analysis because it contains a categorical variable (like zip code) and numerical variables (like square footage, year built, number of bedrooms).

The Problem: ZipCode as Numerical vs Categorical

Key Question: What happens when we treat zipCode as a numerical variable in a decision tree? How does this affect feature importance interpretation?

The Issue: Zip codes (50010, 50011, 50012, 50013) are categorical variables representing discrete geographic areas, i.e. neighborhoods. When treated as numerical, the tree might split on ‚ÄúzipCode > 50012.5‚Äù - which has no meaningful interpretation for house prices. Zip codes are non-ordinal categorical variables meaning they have no inherent order that aids house price prediction (i.e. zip code 99999 is not the priceiest zip code).

Question 1: Numerical vs Categorical Encoding

How should zip code be modeled?

The Consensus: Categorical is Key. It‚Äôs essential to model zip codes as a categorical feature, not numerical. While they look like numbers, they‚Äôre just labels for distinct geographic areas‚Äîthey don‚Äôt represent a quantity, distance, or meaningful rank. Treating them numerically forces the model to invent relationships that simply aren‚Äôt there. For example, 50012 is ‚Äúgreater than‚Äù 50010 in a way that matters for housing prices. In reality, the values are just identifiers.

Is zip code ordinal or non-ordinal?

Ordinality Check: Zip codes are definitively non-ordinal. A higher number (say, 50013) doesn‚Äôt imply a higher price or better neighborhood than a lower number (50010). There‚Äôs no inherent order related to the outcome.

Why does this matter for feature importance?

When a tree treats zip code numerically, it creates nonsensical splits like zipCode <= 50012.5. The model starts grouping neighborhoods based on the first few digits, which has zero real-world meaning for housing prices. This artificial grouping leads to misleading feature importance: the true impact of neighborhood quality gets lost, or the model ends up valuing a meaningless numerical pattern instead. Proper categorical encoding (like one-hot) allows the tree to split on meaningful questions‚Äîe.g., ‚ÄúIs the house in neighborhood A?‚Äù

Question 2: R vs Python Implementation Differences

The Critical Difference

The Critical Difference: Native Handling vs. Manual Encoding The gap between R and Python‚Äôs decision tree implementations comes down to how they handle categorical variables natively. This isn‚Äôt just a minor technicality; it profoundly affects both model efficiency and the resulting interpretability.

What R‚Äôs rpart offers that Python‚Äôs sklearn doesn‚Äôt:

R‚Äôs rpart package supports categorical variables natively. When given a factor variable, the algorithm can test all possible groupings of categories when choosing a split. For four zip codes, rpart can consider splits such as:

*{50010, 50011} vs {50012, 50013}

*{50010} vs {50011, 50012, 50013} and many other combinations

This allows the tree to find the most meaningful way to group neighborhoods based on their relationship to housing prices.

In contrast, scikit-learn‚Äôs DecisionTreeRegressor does not support categorical variables directly. Zip codes must be manually encoded, usually through one-hot encoding. While this avoids the problem of treating them as numeric, it limits the tree to splitting on one category at a time. It cannot create grouped splits the way rpart can, and this can reduce interpretability and model efficiency.

Which implementation is better?

For categorical variables, R‚Äôs rpart handles them more naturally. It produces cleaner trees, more accurate feature importance, and can make more efficient splits because it evaluates category subsets directly.

Documentation from scikit-learn: Scikit-learn‚Äôs documentation explicitly acknowledges the limitation: ‚ÄúAble to handle both numerical and categorical data. However, the scikit-learn implementation does not support categorical variables for now.‚Äù

This is a significant constraint when working with datasets that rely heavily on categorical predictors.

Question 3: Better Categorical Handling in Python

Since scikit-learn does not natively support categorical features in trees, other Python libraries have emerged to handle this better.

CatBoost:

CatBoost is built specifically to work with categorical variables without manual encoding. Its documentation notes:

‚ÄúCatBoost attempts to solve for categorical features using a permutation-driven alternative to the classical algorithm.‚Äù

CatBoost automatically handles categorical variables using ordered target statistics, making it well-suited for problems like modeling neighborhood effects.

LightGBM:

LightGBM also supports categorical variables, though it requires ordinal encoding before training. Once encoded, LightGBM treats categories using an internal ordering strategy that avoids the usual numeric pitfalls.

A common summary from practitioners is:

‚ÄúCatBoost does on-the-fly target encoding, while LightGBM needs ordinal encoding.‚Äù

Both methods avoid strict reliance on one-hot encoding and scale well to larger datasets.

StructureBoost:

StructureBoost is a newer library aimed at categorical variables with more complex structure. An arXiv paper notes:

StructureBoost outperforms established packages such as CatBoost and LightGBM on problems with categorical predictors that contain sophisticated structure.

Practical Recommendation

For most practical applications, CatBoost is the easiest and most reliable way to model categorical variables in Python. It integrates smoothly with common workflows and avoids the limitations of one-hot encoding.
